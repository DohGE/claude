## Checklist
- Only the base-language file (`en.json`) is edited; the other locale files are never touched — no new keys, no English placeholders, no reformatting (the translation team owns them).
- Before adding a key, the base file is searched for an existing translation with the same meaning — generic reusable texts are referenced, never duplicated per area; a new key exists only when no existing one fits semantically; the same phrase never lives under two keys.
- Page-scoped keys follow `page.<area>.labels.<leaf>` / `page.<area>.actions.<leaf>`; global keys follow `<section>.<leaf>` (`labels`, `actions`, `validationMessages`, `chartLabels`, ...); the leaf is descriptive camelCase.
- `<area>` is the parent page/module, never a single component — no new top-level sections and no per-component nodes without a real need; keys shared between areas are reused directly.
- The leaf suffix encodes the UI element type: `…Hint`, `…Toggle`, `…Radio`, `…InputLabel`, `…Selection`, `…Heading`, `…Subheading`, `…Description`, `…Tooltip`, `…SuccessMessage`, `…ErrorMessage`, `…InfoMessage`, `…ConfirmationMessage` — chosen by control type, not by place of use.
- Validation messages go under `validationMessages`; chart labels under `chartLabels` (dynamic backend-driven ones nested); generic button actions in the global `actions` section, view-specific ones under `page.<area>.actions`.
- Values: labels in sentence case without a trailing period; messages as full sentences; `…Heading` in Title Case; interpolation with `{{paramName}}` and the same parameter names used in code.
- Interpolated translations needed in TS are resolved through the translation service's instant lookup with a params object — never assembled by string concatenation.
- Changing or renaming an existing key requires checking every consumer of that key first.
- The JSON stays syntactically valid and stylistically consistent with neighbouring entries; i18n entries have no unit tests.
